Five manager modes for the work agents actually do

The setup conversation can stay free-form, but before the manager starts it resolves to a named recipe or explicit custom mode. Each recipe has settings, permissions, evidence gates, and cleanup rules.

Need overnight progress? Start with GoalBuddy Conveyor and one active board at a time. Need visible quality? Use UX Polish when screenshots should decide the next pass. Need a green merge? Use the PR/CI loop for review, CI monitoring, and merge receipts.

From setup to supervision without losing the thread

Agent Conveyor makes the manager commit to a mode before it gets cut loose, then keeps every loop grounded in Dispatch and receipts.

conveyor manager-config "$TASK" --recipe goalbuddy-conveyor --mode strict conveyor pair --task dogfood-goalbuddy --manager-recipe goalbuddy-conveyor --path /tmp/agent-conveyor-dogfood/workerctl.db conveyor cycle "$TASK" conveyor replay "$TASK"

1
Resolve the recipe

The manager turns free-form intent into `goalbuddy-conveyor`, `ux-polish-loop`, or `custom`.

2
Lock settings

Mode, objective, permissions, tools, epilogues, and cleanup policy are saved in SQLite.

3
Route through Dispatch

Notifications are delivered by push or inbox, with command attempts and side effects recorded.

4
Finish with proof

Completion needs receipts: tests, screenshots, CI, handoff, audit, or an explicit blocker.

5
Dogfood the package

A clean package run launches Dispatch, records acks, satisfies criteria, and finishes without path fixes.

Dogfood proof is a first-class recipe output

A useful manager setup is more than a prompt. It should leave enough evidence for another operator to replay what happened and improve the recipe when friction repeats.

The database is the product feedback loop

Recipe behavior is not hidden in chat memory. The control plane records enough state to replay a run, report a bug, and improve the recipes when dogfooding exposes a weak gate.

manager_configs

Recipe name, supervision mode, permissions, tools, epilogues, and ack policy.

commands

Manager requests, Dispatch claims, retries, blocks, and side-effect risk.

routed_notifications

Worker and manager messages, delivery mode, consumed state, and failures.

acceptance_criteria

Living criteria, deferred follow-ups, satisfied proof, and rejected scope.

manager_cycles

Observation history and the manager context used for each decision.

telemetry_events

Dispatch heartbeat, routing issues, stale work, and searchable issue clues.

When a run feels wrong

Use the database as the bug report, not a reconstructed chat summary.

Start with `conveyor replay`, attach acceptance criteria and notification receipts, then tune the recipe gate that allowed the weak continuation.

Release without token wrangling

Package verification can stay separate from publish. The manual workflow first creates a review tarball; only an explicit publish run with the npm production environment uses Trusted Publishing.

gh workflow run publish.yml -f version=0.1.8 -f publish=false gh workflow run publish.yml -f version=0.1.8 -f publish=true npm view agent-conveyor version
  • Verify first
    Run tests, package smoke, release check, and upload a tarball for review.
  • Publish intentionally
    Require `publish=true`, a matching version input, and the `npm-production` environment.
  • Use OIDC
    Trusted Publishing removes long-lived npm tokens from the release path.
  • Dogfood after publish
    Fresh-install the public package before merging release paperwork.
