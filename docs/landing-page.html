Five manager modes for the work agents actually do

The operator can ask in natural language, but setup resolves to app-visible threads plus a named recipe or explicit custom mode. Each recipe has settings, permissions, evidence gates, and cleanup rules.

Need overnight progress? Start with GoalBuddy Conveyor and one active board at a time. Need visible quality? Use UX Polish when screenshots should decide the next pass. Need a green merge? Use the PR/CI loop for review, CI monitoring, and merge receipts. Need parallel polish? Create a worker set, smoke every app thread, then fan out the UX slices.

From Codex app setup to supervised work

Agent Conveyor now treats live Codex app plumbing as a gate. The operator proves the install, binds visible thread ids, smokes the connection, and only then starts the manager recipe.

conveyor create-disposable-binding "$TASK" --worker-codex-app-thread-id "$WORKER_THREAD_ID" --manager-codex-app-thread-id "$MANAGER_THREAD_ID" --json conveyor app-smoke preflight "$TASK" --mode required --scope pair --json conveyor app-autopilot status "$TASK" --json conveyor replay "$TASK"

Dogfood proof is a first-class operator output

A useful setup is more than a prompt. It should leave enough evidence for another operator to replay what happened, see why real work was allowed, and improve the recipe when friction repeats.

The database is the product feedback loop

Operator behavior is not hidden in chat memory. The control plane records enough state to replay a run, report a bug, and improve the app smoke, wake, and recipe gates when dogfooding exposes a weak continuation.

manager_configs

Recipe name, supervision mode, permissions, tools, epilogues, and ack policy.

app smoke receipts

Smoke ids, nonces, role status, bound thread ids, blockers, and real-work readiness.

routed_notifications

Worker and manager messages, delivery mode, consumed state, and failures.

acceptance_criteria

Living criteria, screenshots, visual-diff proof, deferred follow-ups, and rejected scope.

autopilot policy

Heartbeat automation specs, applied receipts, readiness blockers, quiet-cycle stop recommendations.

telemetry_events

Dispatch heartbeat, routing issues, stale work, wake actions, and searchable issue clues.

When a run feels wrong

Use the database as the bug report, not a reconstructed chat summary.

Start with conveyor replay, attach app-smoke status, autopilot readiness, acceptance criteria, and notification receipts, then tune the gate that allowed the weak continuation.

Release without token wrangling

Package verification can stay separate from publish. The manual workflow first creates a review tarball; only an explicit publish run with the npm production environment uses Trusted Publishing.

gh workflow run publish.yml -f version=0.1.26 -f publish=false gh workflow run publish.yml -f version=0.1.26 -f publish=true npm view agent-conveyor version
  • Verify first
    Run tests, package smoke, release check, and upload a tarball for review.
  • Publish intentionally
    Require publish=true, a matching version input, and the npm-production environment.
  • Use OIDC
    Trusted Publishing removes long-lived npm tokens from the release path.
  • Dogfood after publish
    Fresh-install the public package before merging release paperwork.
